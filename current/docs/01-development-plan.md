# 总体开发方案

## 1. 建设目标

第一阶段先形成最短业务闭环：

> 后台维护品种和示范点内容 → H5 对外展示 → 微信/抖音/二维码传播 → 客户提交咨询或看田意向 → 后台查看线索

第一阶段不追求一次覆盖原方案全部功能。负责人阶段采集、复杂审核、多品种管理和小程序作为后续迭代，以真实运行反馈决定优先级。

## 2. 设计原则

1. **展示优先**：尽快形成能够真实分享的页面。
2. **一个系统**：展示端和管理后台共用一个 Django 项目和数据库。
3. **少依赖**：首版不引入 Redis、消息队列、前后端分离和微服务。
4. **内容可维护**：宣传文字、图片、示范点和联系方式均从后台维护，避免每次修改代码。
5. **逐步发布**：每个 Sprint 都产生可演示、可测试的版本。
6. **保留扩展点**：数据结构按多品种设计，但第一阶段界面只突出一个重点品种。

## 3. 技术架构

### 3.1 技术选型

| 层级 | 技术 | 选择原因 |
|---|---|---|
| 应用框架 | Python 3.13 + Django 5.2 LTS | 管理后台、权限、表单和数据库迁移能力完整，维护成本低 |
| 页面层 | Django Templates + HTML/CSS + 少量 JavaScript/HTMX | 避免单独维护 SPA 和 Node 构建链 |
| 数据库 | PostgreSQL 17 | 稳定、备份工具成熟，适合结构化业务数据 |
| Windows WSGI | Waitress | 原生支持 Windows，部署简单 |
| 反向代理 | Caddy | Windows 可运行，配置较少，可处理 HTTPS 和静态文件 |
| 文件存储 | 首版本地磁盘；上线后可切换对象存储 | 先降低实施复杂度，同时保留云存储接口 |
| 自动测试 | pytest + pytest-django | 便于按功能持续回归 |
| 代码质量 | Ruff | 统一格式及静态检查，执行速度快 |

Django 5.2 是长期支持版本，并支持 Python 3.13。生产环境不能使用 Django 自带的 `runserver`，必须使用正式 WSGI/ASGI 服务；本方案在 Windows 上采用 Waitress。

### 3.2 逻辑结构

```text
微信 / 抖音 / 二维码 / 普通浏览器
                  │ HTTPS
                  ▼
               Caddy
        ┌─────────┴─────────┐
        │                   │
   静态文件/媒体文件     反向代理
                            │
                            ▼
                     Waitress + Django
                       │          │
                       ▼          ▼
                  PostgreSQL   文件存储
```

### 3.3 计划中的 Django 应用划分

| 应用 | 职责 |
|---|---|
| `accounts` | 管理员账号、角色、登录和权限 |
| `varieties` | 品种基础信息、卖点、适宜区域和栽培信息 |
| `sites` | 示范点、位置、联系人、图片及展示状态 |
| `content` | 首页轮播、视频链接、宣传内容和发布状态 |
| `leads` | 咨询、看田、试种、代理等客户线索 |
| `analytics` | 来源参数、页面访问及基础渠道统计 |
| `core` | 公共配置、健康检查、审计字段和通用工具 |

第一阶段不提前建立负责人阶段采集应用；进入第二阶段时再增加 `observations` 或 `collection`，避免首版模型过重。

## 4. 第一阶段功能范围

### 4.1 P0：必须完成

#### 展示端

- 品种首页：主视觉、定位、3—5 个核心卖点。
- 品种详情：审定编号、熟期、株型、适宜区域、密度和栽培要点。
- 示范点列表与详情。
- 田间图片、果穗图片和视频链接展示。
- 咨询、看田、试种表单。
- 联系电话及区域联系人。
- 微信分享元数据与移动端适配。
- 品种二维码、示范点二维码和带来源参数的渠道二维码。
- 隐私提示和表单提交同意。

#### 管理后台

- 管理员登录和基础权限。
- 品种、卖点、示范点和联系方式维护。
- 图片上传、排序、封面设置和发布/下线。
- 视频标题、封面和外部链接维护。
- 客户线索列表、详情和简单状态更新。
- 二维码生成。
- 基础访问量和来源统计。
- 管理操作日志。

### 4.2 P1：试运行后决定

- 视频文件直接上传。
- 数据导出为 Excel。
- 地图定位和导航。
- 销售人员分配与跟进记录。
- 任务提醒。
- 更完整的数据看板。
- 一级内容审核。

### 4.3 P2：后续阶段

- 负责人移动采集和七阶段时间轴。
- 多级内容审核。
- 微信小程序、抖音小程序及挂载能力。
- 多品种管理和品种对比。
- 企业微信集成和完整 CRM。
- 视频转码、剪辑及直播管理。

## 5. 敏捷迭代计划

每个 Sprint 为一周。每周五在测试环境演示，验收通过的功能进入下一次运行环境发布。

| Sprint | 目标 | 主要交付 | 发布范围 |
|---|---|---|---|
| Sprint 0 | 项目准备 | 页面原型、数据模型、Git 规范、环境配置、验收清单 | 不发布业务功能 |
| Sprint 1 | 首页可访问 | 工程基础、首页骨架、移动端布局、健康检查、测试环境首次部署 | 内部访问 |
| Sprint 2 | 内容展示闭环 | 品种详情、示范点列表/详情、后台内容维护 | 内部业务人员录入数据 |
| Sprint 3 | 媒体与发布 | 图片、视频链接、排序、发布/下线、页面性能优化 | 手机端预览 |
| Sprint 4 | 传播与留资 | 微信分享配置、二维码、来源参数、咨询/看田/试种 | 微信小范围试投放 |
| Sprint 5 | 抖音适配与线索 | 抖音内页面适配、线索后台、基础统计、隐私提示 | 抖音企业号/二维码测试 |
| Sprint 6 | 稳定试运行 | 权限、安全、备份恢复、兼容性、操作文档和问题修复 | 正式试运行 |

预计第一阶段为 25—35 个开发人日，日历周期约 5—7 周。平台账号认证、域名备案、素材准备和外部审核等待不计入开发人日。

## 6. 每个 Sprint 的工作方式

### 6.1 Sprint 计划

- Sprint 只设置一个清晰目标。
- 每个开发任务控制在 0.5—2 天内。
- 每个任务必须写验收条件和权限边界。
- 未验收完成的任务不能以“开发完成”计入进度。

### 6.2 Codex 协作约定

- 一次只实现一个可验证的小功能。
- Codex 修改前先读取需求、已有代码及项目约定。
- 每项修改同时补充测试和必要文档。
- Codex 运行格式检查、测试和部署检查后再交付。
- 人工负责业务判断、页面效果确认、账号权限和真实数据验收。
- 不允许把真实密钥、数据库密码和客户手机号提交到 Git。

项目开始编码时，应在仓库根目录建立 `AGENTS.md`，记录目录规范、测试命令、迁移规则、权限原则、部署命令和禁止事项。

## 7. 分支与发布策略

首版采用简单的主干开发：

- `main`：任何时刻都应可部署。
- 功能分支：`feature/<简短名称>`。
- 修复分支：`fix/<简短名称>`。
- 发布标签：`v0.1.0`、`v0.2.0` 等。

功能合并前至少满足：

1. 自动测试通过。
2. 数据库迁移文件已检查。
3. 不包含密钥和真实隐私数据。
4. 在测试环境完成关键路径操作。
5. 有明确回滚方案。

## 8. 测试策略

| 测试类型 | 重点内容 | 执行时机 |
|---|---|---|
| 单元测试 | 模型校验、状态转换、来源识别 | 每次提交 |
| 集成测试 | 登录权限、后台保存、表单提交、文件上传 | 每个功能完成时 |
| 页面测试 | 首页、品种、示范点、线索全流程 | 每个 Sprint |
| 移动端测试 | 微信内置浏览器、抖音内置浏览器、常用安卓/iPhone | 每次发布前 |
| 安全检查 | DEBUG、密钥、权限、上传类型、HTTPS、CSRF | 测试及运行环境发布前 |
| 恢复测试 | 数据库备份和文件恢复 | 正式试运行前及每季度 |

最低支持目标：近两年的主流 Android 微信、iOS 微信、Android 抖音、iOS 抖音，以及桌面 Chrome/Edge。

## 9. 上线门槛

进入运行环境前必须满足：

- 正式域名已解析并启用 HTTPS。
- `DEBUG=False`，生产密钥未进入源代码。
- `python manage.py check --deploy` 无阻断问题。
- 数据库已备份，恢复步骤经过验证。
- 管理后台不能被匿名访问。
- 上传文件限制了类型和大小。
- 表单有隐私说明、必填校验和防重复提交措施。
- 首页、示范点、咨询表单在微信和抖音内完成实机测试。
- 发布后能够通过健康检查和关键业务冒烟测试。

## 10. 主要风险与应对

| 风险 | 应对方式 |
|---|---|
| 微信/抖音平台能力与账号权限不确定 | 第一阶段以 HTTPS H5、企业号引导和二维码为主；小程序挂载单独立项 |
| 图片过大导致移动端加载慢 | 后台上传时压缩图片、生成缩略图并限制单文件大小 |
| 视频导致服务器带宽和磁盘压力 | 第一阶段优先使用审核后的视频链接；上传能力后置 |
| 内容频繁变化导致反复改代码 | 所有宣传内容和排序均从后台维护 |
| Windows 服务重启后系统未启动 | Waitress 和 Caddy 注册为自动启动的 Windows 服务 |
| 数据丢失 | 数据库与媒体文件分别备份，并定期做恢复演练 |

## 11. 第一阶段验收结果

第一阶段完成后应达到：

1. 业务人员无需改代码即可维护品种、卖点、示范点、图片和视频链接。
2. 客户可通过微信、抖音引导和二维码访问适配手机的 H5 页面。
3. 客户可以提交咨询、看田或试种，后台能够查看和处理线索。
4. 系统能够区分主要渠道来源并提供基础统计。
5. 测试和运行环境均可在 Windows 上重复安装、发布、备份和恢复。
