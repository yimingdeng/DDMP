# 玉米重点品种数字展示平台：详细功能需求与迭代基线

## 文档信息

| 项目 | 内容 |
|---|---|
| 文档版本 | v1.0 |
| 基线日期 | 2026-06-30 |
| 当前状态 | 第一阶段开发基线 |
| 适用范围 | H5 展示平台、基础管理后台、线索与基础统计 |
| 开发方式 | 一周一个 Sprint，逐步上线验证 |
| 技术基线 | Django 单体应用、PostgreSQL、Windows |

本文档是后续需求拆分、Codex 开发、测试和验收的共同依据。开发任务应引用功能编号，例如“实现 `FE-SITE-002` 示范点详情页”，避免仅使用模糊自然语言描述。

## 1. 产品目标与边界

### 1.1 第一阶段目标

第一阶段形成以下闭环：

> 管理员维护重点品种和示范点 → 发布 H5 展示内容 → 通过微信、抖音引导和二维码传播 → 客户查看内容并提交咨询 → 后台查看、标记和导出线索

### 1.2 第一阶段成功标准

1. 业务人员无需修改代码即可维护一个重点品种及 3—5 个示范点。
2. 微信、抖音内置浏览器和普通手机浏览器均可正常浏览。
3. 品种、示范点、图片、视频链接和联系方式可以独立发布或下线。
4. 客户能够完成咨询、看田或试种表单提交。
5. 后台能够查看线索，并区分微信、抖音、二维码等来源。
6. 测试和运行环境可以按照现有 Windows 文档重复部署、备份和恢复。

### 1.3 第一阶段明确不做

- 不开发微信小程序和抖音小程序。
- 不承诺抖音短视频或直播锚点挂载。
- 不开发负责人七阶段采集端。
- 不开发多级内容审核。
- 不实现复杂销售 CRM、合同、订单和支付。
- 不实现视频转码、剪辑、直播推流和视频文件托管。
- 不做微服务、Redis、消息队列和独立前端 SPA。
- 不做多租户；系统仅服务一个公司主体。

### 1.4 优先级定义

| 优先级 | 定义 | 处理规则 |
|---|---|---|
| P0 | 第一阶段上线不可缺少 | 必须开发并验收 |
| P1 | 试运行价值较高 | 由真实反馈决定进入后续 Sprint |
| P2 | 第二阶段及以后 | 不影响第一阶段上线 |

## 2. 用户角色

### 2.1 第一阶段角色

| 角色代码 | 角色 | 说明 |
|---|---|---|
| `VISITOR` | 外部访客 | 匿名浏览公开页面、提交意向表单 |
| `CONTENT_EDITOR` | 内容管理员 | 维护品种、卖点、示范点、媒体和联系人 |
| `LEAD_OPERATOR` | 线索处理人员 | 查看和更新客户线索，不修改展示内容 |
| `SYSTEM_ADMIN` | 系统管理员 | 管理账号、权限、系统配置和所有业务数据 |

小团队可以让同一自然人同时拥有多个后台角色，但必须使用个人账号，不使用共享管理员账号。

### 2.2 第二阶段预留角色

| 角色代码 | 角色 | 说明 |
|---|---|---|
| `SITE_OWNER` | 示范点负责人 | 采集本人负责示范点的阶段数据 |
| `REGIONAL_REVIEWER` | 区域管理员 | 审核本区域示范点提交内容 |
| `VARIETY_MANAGER` | 品种管理员 | 审核并决定数据是否公开 |
| `MANAGEMENT_VIEWER` | 管理层 | 查看进度和推广数据，无编辑权限 |

### 3.1 对象关系

```text
品种 1 ── N 核心卖点
品种 1 ── N 示范点
品种/卖点/示范点 1 ── N 媒体素材
示范点 N ── N 联系人
品种/示范点 1 ── N 客户线索
来源渠道 1 ── N 访问事件
来源渠道 1 ── N 客户线索
品种/示范点 1 ── N 二维码
```

虽然第一阶段只突出一个重点品种，数据表必须保留品种主键关系，避免第二个品种上线时重构数据库。

## 4. 状态与业务规则

### 4.1 展示内容状态

品种、示范点、卖点和媒体统一采用：

```text
DRAFT 草稿 → PUBLISHED 已发布 → ARCHIVED 已归档
                ↑        ↓
                └── 重新发布
```

规则：

- 草稿只能在后台查看。
- 已发布内容可以被匿名访客查看。
- 已归档内容保留历史数据，但不出现在公开列表。
- 后台普通删除采用软删除或归档；存在引用的数据不能直接物理删除。
- 发布必须记录发布时间和发布操作人。

### 4.2 示范点开放状态

示范点另有参观状态：

| 状态 | 含义 | 前端行为 |
|---|---|---|
| `NOT_OPEN` | 暂不开放看田 | 不显示预约按钮或提示暂未开放 |
| `OPEN` | 可预约看田 | 显示看田预约按钮 |
| `CLOSED` | 已结束 | 显示展示内容，但提示活动已结束 |

### 4.3 线索状态

```text
NEW 新线索 → CONTACTING 跟进中 → COMPLETED 已完成
    └──────────────→ INVALID 无效
```

第一阶段规则：

- 新提交线索默认状态为 `NEW`。
- 线索处理人员可以改为跟进中、已完成或无效。
- 每次状态变化记录操作人、时间和备注。
- 第一阶段不实现复杂销售机会阶段和自动分配。
- 线索默认不允许物理删除；测试数据由系统管理员标记为无效。

### 4.4 来源归因

标准来源代码：

| 来源代码 | 名称 |
|---|---|
| `wechat_moments` | 微信朋友圈 |
| `wechat_channels` | 微信视频号 |
| `wechat_group` | 微信群 |
| `official_account` | 微信公众号 |
| `douyin_video` | 抖音短视频 |
| `douyin_live` | 抖音直播 |
| `douyin_profile` | 抖音企业号主页 |
| `field_qrcode` | 田间二维码 |
| `direct` | 无来源参数直接访问 |
| `other` | 其他来源 |

归因规则：

1. 链接使用 `source=<来源代码>`。
2. `source` 只表示传播渠道或入口位置；销售人员、经销商、区域经理等身份后续由后台用户所属组或绑定身份识别，不再作为传播渠道代码。
3. 有合法来源参数时，记录当前访问来源，并在浏览器保存 30 天。
4. 客户提交表单时，线索记录最近一次合法来源。
5. 没有来源参数和历史来源时记为 `direct`。
6. 非法或超长来源参数不直接入库，统一记为 `other`。
6. 第一阶段不做跨设备追踪和设备指纹。

## 6. 展示端详细功能

### 6.1 全局框架

#### FE-GLOBAL-001：响应式页面框架

- 优先级：P0
- Sprint：Sprint 1
- 功能：提供统一页头、页脚、内容宽度、颜色、按钮、字体、加载和错误样式。
- 规则：移动端为主要设计目标；桌面端不能出现横向滚动或内容过度拉伸。
- 测试预览：公开展示页面支持在地址后添加 `?mobile=1`，在桌面浏览器中强制使用手机宽度和移动端响应式样式；页面内跳转继续保持预览模式。
- 验收：常见 Android/iPhone 屏幕、微信/抖音内置浏览器和桌面 Edge/Chrome 可正常浏览。

#### FE-GLOBAL-002：全局导航

- 优先级：P0
- Sprint：Sprint 1
- 功能：首页、品种详情、示范点、咨询入口。
- 规则：手机端使用简洁菜单或底部操作区；核心咨询按钮在关键页面易于发现。
- 手机底部菜单：首页显示“看详情、我要咨询、看示范”；品种、卖点、示范点列表和示范点详情页第一项统一为“返回”，其余操作按当前页面上下文显示。
- 咨询入口：优先定位到区域联系人，用户继续下滑可填写咨询表单；页头已有联系入口时，底部不重复显示“联系销售”。
- 显示范围：底部菜单在不大于 860 px 的手机或小屏设备显示，后台、登录、田间采集、隐私和表单操作页不显示。
- 验收：访客最多两次点击可从任一主要页面进入咨询或示范点列表。

#### FE-GLOBAL-003：站点页脚

- 优先级：P0
- Sprint：Sprint 1
- 展示：公司名称、联系电话、版权、隐私说明入口、必要的备案信息。
- 规则：所有内容由后台站点配置维护。
- 验收：修改后台配置后无需发布代码即可更新。

#### FE-GLOBAL-004：错误和空状态

- 优先级：P0
- Sprint：Sprint 3
- 功能：404、500、内容未发布、列表为空、网络提交失败等统一提示。
- 规则：错误页面不能暴露代码、数据库和服务器路径。
- 验收：访问不存在或已归档内容时返回正确状态码及友好页面。

### 6.2 首页

#### FE-THEME-001：公开营销主题配置

- 优先级：P1
- 管理员可在“站点配置 → 系统主色调”选择“系统基本色调（绿色）”或“紫色＋黄色”。
- 配置作用于公开营销首页、示范阶段传播页以及之后重新生成的朋友圈海报和短视频封面。
- 业务采集端和 Django 后台管理端不随公开营销主题切换。
- 主题切换不得修改业务数据；已生成的营销图片通过后台重新生成后应用新主题。

#### FE-HOME-001：首屏主视觉

- 优先级：P0
- Sprint：Sprint 1
- 展示：品种名称、主视觉图、一句话定位、主行动按钮。
- 主行动按钮：查看示范点、我要咨询，可后台配置显示文字。
- 重点单品种营销首页颜色由站点主题配置决定，主题样式必须限定在公开营销页面，不影响采集端和后台管理端。
- 验收：后台替换主视觉和定位后前端自动更新；未发布品种不显示。

#### FE-HOME-002：核心卖点区

- 优先级：P0
- Sprint：Sprint 2
- 展示：3—5 个已发布卖点的标题、简述、图片和详情入口。
- 移动端：卖点卡片使用紧凑字号、间距和两行摘要，避免单张无图卡片占据大半屏幕。
- 排序：按后台排序值升序，相同排序按创建时间。
- 验收：后台调整顺序、发布或归档后前端列表正确变化。

#### FE-HOME-003：最新示范点区

- 优先级：P0
- Sprint：Sprint 2
- 展示：示范点名称、地区、封面、当前阶段摘要、主要表现。
- 规则：首页展示后台标记为推荐且已发布的前 N 个示范点，N 默认 3。
- 验收：未发布和已归档示范点不得出现。

#### FE-HOME-004：图片与视频入口

- 优先级：P0
- Sprint：Sprint 3
- 展示：精选田间图片、果穗图片和视频封面。
- 规则：视频第一阶段使用审核后的链接；点击行为必须清晰，不自动有声播放。
- 验收：没有视频时整个区块可隐藏，不出现空白容器。

#### FE-HOME-005：转化入口

- 优先级：P0
- Sprint：Sprint 4
- 展示：我要咨询、我要看田、我要试种三个入口。
- 验收：每个入口打开相应意向类型的表单，并继承当前品种、示范点和来源。

### 6.3 品种详情

#### FE-VAR-001：品种基本信息

- 优先级：P0
- Sprint：Sprint 2
- 展示：名称、审定编号、适宜区域、熟期、株型、穗型、籽粒类型、适宜密度。
- 规则：空字段不显示空标签；长文本按段落展示。
- 验收：页面内容与后台已发布版本一致。

#### FE-VAR-002：栽培要点与风险提示

- 优先级：P0
- Sprint：Sprint 2
- 展示：播期、密度、水肥管理、栽培要点和风险提示。
- 规则：风险提示不得通过前端样式隐藏或弱化到不可见。
- 验收：风险提示存在时必须在品种详情正文中可见。

#### FE-VAR-003：卖点详情

- 优先级：P0
- Sprint：Sprint 3
- 展示：卖点标题、说明、关联图片、数据说明和关联示范点。
- 移动端：采用紧凑标题区，图片和视频优先于长文字展示；画廊使用紧凑双列布局，正文、视频卡片及数据说明减少字号和留白但保留完整内容。
- 规则：第一阶段的数据说明为人工审核后录入的展示文字，不自动从原始试验数据计算。
- 验收：未发布媒体不展示，已归档卖点返回不可公开状态。

#### FE-VAR-004：移动端推广信息层级

- 优先级：P0
- Sprint：移动推广优化
- 首屏：封面图片或视频优先，紧接品种名称、一句话定位、适宜区域、最多 3 个短标签和咨询/看田入口。
- 顺序：核心优势、视频、图片、关键参数、关联示范点、折叠的品种简介与栽培资料；风险提示保持可见，不得折叠隐藏。
- 展示：移动端整体采用简洁、紧凑的信息密度；核心优势卡片使用小尺寸缩略图或编号配合标题、两行摘要纵向连续展示，不使用占据大半屏幕的纯编号卡片。
- 操作：手机端常驻“联系销售、我要咨询、我要看田”底部操作栏，主要转化操作一次点击可达。
- 内容：卖点使用图片、短标题和短说明；参数使用两列数据卡片；长篇介绍和栽培说明按需展开。
- 性能：首屏封面使用缩略图并优先加载，非首屏图片懒加载，视频只预加载元数据且不得自动有声播放。
- 验收：宽度 360—430 px 时无横向页面滚动；首屏可识别品种、适宜区域和核心定位；底部操作不遮挡表单及页脚。

### 6.4 示范点

#### FE-SITE-001：示范点列表

- 优先级：P0
- Sprint：Sprint 2
- 展示：封面、示范点名称、省市县、当前阶段、主要表现、参观状态。
- 移动端：地图页减少标题与控件留白；切换列表后使用小缩略图、标题和两行摘要组成的紧凑横卡，避免大图卡片占满首屏。
- 筛选：第一阶段支持按省份或区域筛选；数据量少时允许折叠或暂不显示筛选器。
- 验收：只有已发布示范点进入列表；筛选结果准确且可清除。

#### FE-SITE-002：示范点详情

- 优先级：P0
- Sprint：Sprint 2
- 展示：名称、区域、地址、播种日期、面积、密度、种植模式、当前阶段、介绍、图片和联系人。
- 移动端：压缩详情首屏和区块间距，概况数据使用紧凑双列卡片，标题、介绍和看田服务保持清晰但不使用桌面端的大字号与大留白。
- 隐私：公开地址允许精确到乡镇/村，但是否显示详细门牌由后台控制。
- 验收：所有公开字段均受后台公开开关控制，不展示内部备注。

#### FE-SITE-003：看田状态和预约

- 优先级：P0
- Sprint：Sprint 4
- 规则：`OPEN` 显示预约按钮；`NOT_OPEN` 提示暂未开放；`CLOSED` 提示已结束。
- 验收：状态切换后前端按钮及提示同步变化。

#### FE-SITE-004：地图与导航

- 优先级：P1
- Sprint：试运行后安排

## 7. 管理后台详细功能

### 7.1 登录、账号与权限

#### ADM-AUTH-001：后台登录

- 优先级：P0
- Sprint：Sprint 1
- 功能：用户名/密码登录、退出、登录失败提示。
- 安全：密码使用 Django 安全哈希；失败提示不区分账号不存在和密码错误。
- 登录规则：账号必须是后台人员且至少拥有一个已注册后台模块权限，或为超级管理员；没有后台模块权限的人员在登录时直接提示，不进入后台首页。
- 退出规则：后台管理、负责人采集和营销发布中心各自退出到各自登录页；如果用户从公开首页底部入口进入“我的示范”或“我要发布”，退出后返回公开首页。
- 菜单规则：登录后的顶部菜单、筛选项和快捷入口必须按实际服务端权限显示；无权限访问的功能不得仅靠进入后提示错误来控制。
- 验收：匿名访问后台自动跳转登录；退出后原后台页面不可通过返回继续操作；不同子系统退出不得统一跳到负责人采集登录页。

#### ADM-AUTH-002：账号管理

- 优先级：P0
- Sprint：Sprint 2
- 功能：系统管理员创建、停用账号并分配角色。
- 规则：不提供公开注册；账号停用后会话应在合理时间失效；后台用户信息必须维护“手机”，该字段必填且唯一。
- 登录名：同一套账号支持使用账号、邮箱或手机号任一方式登录后台管理、负责人采集和营销发布中心。
- 迁移：现有用户在升级时先自动生成一个临时手机号，管理员后续可在后台修改为真实手机号。
- 姓名：用户个人信息按中文习惯以“姓在前、名在后”的顺序输入和展示。
- 验收：非系统管理员无法管理账号和权限；手机号缺失或重复时不能保存用户；使用账号、邮箱、手机号均可登录同一账号。

#### ADM-AUTH-003：修改密码

- 优先级：P0
- Sprint：Sprint 2
- 功能：登录用户修改自己的密码，系统管理员重置密码。
- 验收：修改后旧密码失效；日志不得记录密码内容。

## 9. 系统级功能

#### SYS-HEALTH-001：健康检查

- 优先级：P0
- Sprint：Sprint 1
- 功能：提供应用和数据库健康状态，用于部署验证。
- 安全：公网响应只返回简单状态，不暴露数据库地址、版本和异常堆栈。
- 验收：应用正常返回 200；数据库不可用时返回非 200 并写入内部日志。

#### SYS-CONFIG-001：环境配置隔离

- 优先级：P0
- Sprint：Sprint 0—1
- 规则：开发、测试、运行使用不同数据库、密钥、域名和媒体目录。
- 验收：仓库不包含真实 `.env`；测试环境不能连接运行数据库。

#### SYS-SEC-001：生产安全设置

- 优先级：P0
- Sprint：Sprint 6
- 内容：`DEBUG=False`、HTTPS、安全 Cookie、限定主机、CSRF、防点击劫持和安全响应头。
- 验收：`python manage.py check --deploy` 无阻断问题。

#### SYS-FILE-001：文件安全

- 优先级：P0
- Sprint：Sprint 3
- 规则：限制扩展名、MIME、大小和图片尺寸；媒体目录不执行脚本；文件名由系统生成。
- 验收：伪装脚本、超大文件和损坏图片被拒绝。

#### SYS-LOG-001：应用日志

- 优先级：P0
- Sprint：Sprint 1—6
- 内容：启动、错误、关键后台操作、表单失败和安全事件。
- 规则：不写密码、密钥、Cookie、数据库连接串和完整客户隐私数据。
- 验收：能够根据请求时间和请求 ID 定位一次失败，但日志本身不泄密。

#### SYS-BACKUP-001：备份恢复

- 优先级：P0
- Sprint：Sprint 6
- 内容：数据库每日备份、媒体增量备份、异机保留和恢复演练。
- 验收：测试环境成功恢复一次数据库和媒体，并记录耗时和结果。

#### SYS-ERROR-001：异常处理

- 优先级：P0
- Sprint：持续
- 规则：用户看到友好提示；后台日志保留错误详情；生产页面不显示堆栈。
- 验收：模拟 404、数据库异常和表单异常时行为符合要求。

## 10. 非功能需求

### 10.1 性能

第一阶段初始目标，可在 Sprint 0 根据服务器配置调整：

- 手机 4G 网络下，首页主要内容力争 3 秒内可见。
- 普通动态页面服务器响应时间在正常负载下力争小于 1 秒，不包含外部视频平台耗时。
- 图片必须提供适合列表的缩略图，首屏外图片懒加载。
- 首页不加载不必要的大视频文件和第三方脚本。
- 初始按 50 个并发请求和每日 5,000 次页面访问进行基础验证。

性能目标是上线验收目标，不是无限容量承诺；实际访问增长后根据日志扩容。

### 10.2 可用性和兼容性

- 支持近两年的主流 Android/iPhone 系统中的微信和抖音内置浏览器。
- 支持当前稳定版和前一个主版本的桌面 Edge/Chrome。
- 关键表单不能依赖悬停、右键或仅桌面可见操作。
- 外部视频失效时不能阻塞页面其他内容。

### 10.3 安全和隐私

- 后台和表单全站 HTTPS。
- 采用个人后台账号、最小权限和强密码。
- 客户手机号、地址和备注按敏感数据处理。
- 列表默认脱敏手机号，详情查看受权限控制并记录日志。
- 隐私数据不进入统计事件和普通应用日志。
- 数据导出、批量查看和删除属于受控操作。
- 数据保留期限在正式上线前由公司确定，并写入隐私说明和清理任务。

### 10.4 可维护性

- 单体 Django 项目，一个代码仓库、一套迁移历史。
- 业务规则和字段校验应有自动测试。
- 所有依赖锁定版本并定期升级，不在服务器临时安装未知包。
- 配置从环境读取，代码中不硬编码域名、密码和绝对生产路径。
- 每次 Sprint 同步更新本功能文档、变更记录和操作文档。

### 10.5 时间和时区

- 系统业务时区统一为 `Asia/Shanghai`。
- 数据库存储策略由工程统一实现，不允许不同模块混用无时区时间。
- 页面显示到分钟；日志和审计保留到秒或更高精度。

### 11.1 负责人端

- 优先级：P2

#### COL-TASK-001：我的示范点和待办

- 优先级：P2

#### COL-DRAFT-001：采集草稿

- 优先级：P2

- 规则：草稿不进入审核和公开页面。

#### COL-SITE-001：示范点基本信息与定位

- 优先级：P1

- 规则：经纬度必须同时填写或同时留空，服务端校验合法范围；修改成功写入审计日志，未授权访问返回 404。

#### COL-PHOTO-001：现场图片上传

- 优先级：P2
- 功能：手机拍照/相册上传、压缩、说明、必传校验。
- 规则：记录上传人和时间；是否保留 GPS/EXIF 需由隐私要求决定。

#### COL-ANOMALY-001：异常上报

- 优先级：P2
- 字段：异常类型、程度、发生日期、说明、图片和建议措施。

#### COL-HISTORY-001：提交记录

- 优先级：P2

### 11.2 七阶段采集字段

以下是第二阶段默认固定字段。上线前再决定哪些数值字段必须使用统一单位和范围。

#### 播种阶段

- 苗期照片必传。

#### 拔节期

- 拔节日期、长势、株型、整齐度、叶片表现。
- 病虫害及程度、追肥/浇水/除草等管理措施。
- 拔节期照片必传。

#### 抽雄吐丝期

- 抽雄日期、吐丝日期、花期协调性、雄穗大小、花粉量。

#### 灌浆期

- 灌浆期照片必传。

#### 成熟期

- 成熟日期、株高、穗位、倒伏率、倒折率、空秆率、穗腐率。
- 青枯情况、保绿性、脱水速度、站秆能力。
- 果穗整齐度、籽粒商品性。
- 成熟期照片必传，视频建议上传或填写链接。

#### 收获期

- 收获日期、收获方式、实收面积、实收重量、籽粒含水率。
- 折合亩产由系统根据正式公式计算，并保留原始输入。
- 机收破碎、落粒、顺畅度和商品性评价。
- 测产照片必传，视频建议上传或填写链接。

### 11.3 审核与公开

第二阶段建议状态：

```text
DRAFT 草稿
  → SUBMITTED 已提交
  → REGIONAL_APPROVED 区域通过
  → HQ_APPROVED 总部通过
  → PUBLISHED 已公开

SUBMITTED / REGIONAL_APPROVED / HQ_APPROVED
  → REJECTED 已退回 → 修改后重新提交
```

#### REV-001：区域审核

- 查看原始数据和媒体，填写通过或退回意见。
- 只能审核本区域示范点。

#### REV-002：总部审核

- 在区域通过后确认展示内容和公开范围。
- 可以只公开部分审核字段或使用单独的展示摘要。

#### REV-003：内部数据与展示数据隔离

- 原始记录不可因公开文案调整而覆盖。
- 对外展示内容保存审核来源、版本和发布时间。

#### REV-004：审核历史

- 保存每次提交、通过、退回、发布的操作人、时间和意见。
- 审核历史不可由普通用户修改。

### 11.4 第二阶段看板

#### DASH-COL-001：采集进度

- 按示范点和阶段展示应采、已采、审核中、退回和已完成数量。

#### DASH-COL-002：数据完整度

- 统计必填字段、必传照片和阶段完成率。

#### DASH-COL-003：异常汇总

- 按区域、类型和程度汇总倒伏、病害、缺苗等异常。

## 13. 第一阶段端到端验收场景

### E2E-001：内容发布

1. 内容管理员登录后台。
2. 新建品种草稿、卖点和示范点。
3. 上传图片并设置封面。
4. 通过预览检查内容。
5. 发布品种和示范点。
6. 匿名访客从公开地址查看。

预期：草稿阶段匿名不可见；发布后内容、顺序和图片正确；内部备注不出现在页面源码。

### E2E-002：微信来源咨询

1. 使用 `source=wechat_moments` 的二维码或链接进入首页。
2. 浏览品种和示范点。
3. 提交“我要咨询”。
4. 线索处理人员登录后台查看。

预期：线索包含正确品种、来源、落地页、隐私版本和状态 `NEW`；重复刷新不重复创建。

### E2E-003：看田预约状态

1. 后台将示范点参观状态设置为开放。
2. 访客进入示范点并提交看田申请。
3. 后台将状态改为已结束。

预期：开放时显示预约；已结束后停止新预约并显示结束提示；历史线索保留。

### E2E-004：权限隔离

1. 内容管理员尝试访问线索列表和详情。
2. 线索人员尝试修改品种和站点配置。
3. 匿名用户直接访问后台 URL。

预期：所有越权访问均被拒绝并留下必要安全日志，不因隐藏菜单而代替服务端权限校验。

### E2E-005：内容下线

1. 已发布示范点生成渠道二维码并产生访问记录。
2. 后台归档该示范点。
3. 再次扫码。

预期：不显示已归档详情，展示友好提示或品种主页入口；历史访问和二维码记录不丢失。

### E2E-006：备份恢复

1. 备份数据库和媒体目录。
2. 在测试环境创建空数据库和空媒体目录。
3. 执行恢复。
4. 登录后台并验证首页、示范点、图片和线索。

预期：核心数据和媒体引用完整，过程有记录且不影响运行环境。

- [ ] 哪些联系人姓名和手机号允许公开。
- [ ] 首批田间图片和视频链接已经取得使用授权。
- [ ] 测试域名、正式域名以及微信/抖音相关企业账号负责人。
- [ ] 客户线索由谁查看、谁跟进、多久处理。
- [ ] 隐私说明负责人、客户数据保留期限和删除流程。
- [ ] 微信、抖音、Android、iPhone 的首批验收人员和测试设备。
- [ ] 首页原型、主色、字体和宣传口径获得业务确认。
- [ ] 第一阶段范围和“不做清单”获得确认。

某项资料未准备好时，可以使用明确标注的测试内容完成工程验证，但不得把测试文字、无授权图片或虚构联系方式发布到正式环境。
- 展示顺序：首页配置了封面图片或视频时，媒体内容必须作为首屏第一个内容展示，之后再显示品种名称、定位和操作入口。
- 后台内联：在品种、核心卖点或示范点编辑页新增媒体时，系统必须使用当前父对象建立关联，不得因表单校验阶段的空关联缓存误报“关联对象不存在”。
- 图片兼容：支持 JPEG、PNG、WebP，以及手机多镜头照片使用的 JPEG 兼容 MPO 文件；扩展名与真实格式不一致时应显示检测到的格式，不得仅提示笼统错误。
- 阶段记录：按生育进程倒序展示（收获至播种）；每个阶段先展示已审核公开的照片和视频，再展示公开摘要与详细数据；现场照片可点击进入大图灯箱并在同阶段照片间切换。
- 移动灯箱：缩略图打开后使用近全屏空间放大展示原图；关闭和左右切换按钮固定在屏幕安全区内，不得因图片尺寸或滚动移出视口。
- 首页咨询状态：用户从首页进入“我要咨询”或“联系我们”区域后，移动端底部左按钮必须显示“返回”并返回首页顶部，不得继续显示“看详情”。
- 类型切换：后台将媒体切换为“本地视频”时，自动清空遗留的普通图片、视频平台和外部链接，保留可用的视频封面；不得要求录入人员手工清除隐藏的旧值。
- 类型切换：切换为首页或咨询二维码时自动清空旧品种和示范点；切换为品种目标时清空旧示范点，切换为示范点目标时清空旧品种，避免旧字段残留导致保存失败。
- 当前阶段：以最高已发布田间阶段为公开下限；发布新阶段时自动推进，补发较早阶段不得导致当前阶段倒退，后台也不得保存低于已公开阶段的值。
- 展示顺序：后台列表直接编辑“展示顺序”，数字越小越靠前；公开示范点列表在无法获取用户位置时使用该顺序兜底，地图数据和首页示范点入口继续使用后台展示顺序。
